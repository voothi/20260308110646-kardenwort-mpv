## Context

In the Kardenwort-mpv engine, paired multiselect ("Pink" selection) is stored in `FSM.DW_CTRL_PENDING_SET`. When a word is toggled into this set via `f` or `Ctrl+LMB` in Drum Window mode, `dw_osd:update()` is correctly called. However, in Drum Mode (windowless) and SRT mode, the visual update is not triggered because `ctrl_toggle_word` hardcodes a check only for `FSM.DRUM_WINDOW ~= "OFF"`.

Additionally, in windowless modes, the OSD content (`drum_osd.data`) is generated by `tick_drum`, which relies on `has_pink` logic in `master_tick`. To provide immediate visual feedback upon `f`, the toggle action must trigger a redraw of the drum/SRT OSD.

## Goals / Non-Goals

**Goals:**
- Ensure paired selections (Pink) update immediately in Drum Mode and SRT Mode.
- Refactor `ctrl_toggle_word` (or the handlers) to correctly invalidate or update `drum_osd` when outside of Drum Window.

**Non-Goals:**
- Do not change how Pink multiselect logic fundamentally works.
- Do not alter Drum Window behavior.

## Decisions

- **Update UI correctly in `ctrl_toggle_word`**: Modify `ctrl_toggle_word` in `scripts/lls_core.lua` to check if it's in Drum Mode or SRT mode. To provide immediate feedback, `ctrl_toggle_word` (or `cmd_dw_toggle_pink`) should call `force_ui_update()` or `master_tick()` to regenerate the OSD data (`drum_osd.data`) and push it to the screen. 

## Risks / Trade-offs

- **Performance**: Calling a redraw on every word toggle is negligible since it only happens on explicit user input.
