# RFC: Release v1.26.24 (Isotropic Mouse Hit-Testing)

**Request ZID**: 20260322154532
**Date**: 2026-03-22
**Status**: DRAFT → PROPOSED

## Overview
This release addresses a critical text selection bug where hit-test alignment completely drifted when the mpv window was resized or snapped to non-16:9 aspect ratios (e.g., half-screen). It fundamentally fixes the mathematical model that translates physical mouse coordinates into the virtual subtitle tracking grid.

## Included Change Requests (ZIDs)
- **20260322153811**: Investigation into hit-testing accuracy failure in snapped/resized windows.
- **20260322154532**: Formal release and documentation of the Isotropic scaling fix.

## Implementation Details

### 1. The Aspect Ratio Drift Problem (Analytics)
The Drum Window's hit-testing engine (`dw_get_mouse_osd`) originally determined the virtual X-coordinate of the mouse by scaling the absolute window width (`mx / ow * 1920`). However, the `libass` subtitle renderer preserves text aspect ratios. It scales text isotropically based purely on the window's *height*, without squishing text to fit a narrower window. Because the text wasn't squishing but the math assumed it was, every physical pixel moved by the mouse counted as *double* the distance in the hit-test engine when the window was half-width.

### 2. Isotropic Coordinate Mapping (Decision)
The coordinate resolution mapping was entirely rewritten to use an `isotropic` scaling factor (`scale_isotropic = oh / 1080`).
- **Y-Axis**: Continues to scale directly against the physical window height.
- **X-Axis**: The mathematical anchor was explicitly set to the physical center of the screen `(ow / 2)`. The horizontal offset is now translated purely using the height-derived scaling factor, exactly mirroring `libass` rendering behavior.

This mathematically guarantees that the hit-test grid tracks the physical pixels of the text flawlessly, completely irrespective of window stretching, letterboxing, or snapping.
